## 项目本质

Matt Pocock 的 `skills` 是一个**技能集合**——从他的 `.claude` 目录直接导出，任何人都可以 clone 使用。

核心哲学：**技能不是 prompt，是可执行的代码**。

示例结构：
```
skills/
├── git/
│   ├── squash-commits.sh      # 压缩 git 提交
│   └── find-branch.sh         # 查找分支
├── code/
│   ├── extract-interface.sh   # 提取接口
│   └── add-types.ts           # 添加类型
└── ...
```

每个 skill 都是一个 Shell 脚本，通过 `claude code` 命令调用。

---

## 为什么火

1. **零学习成本** — 不需要理解什么"工程化 prompt"，直接 clone 用
2. **可调试** — 是代码，不是魔法 prompt，可以看、可以改
3. **可组合** — 多个 skills 叠加，形成工作流
4. **Matt Pocock 的影响力** — 他在 Twitter/X 上的推广能力很强

---

## 对比：Skills vs Traditional Prompts

| 维度 | Traditional Prompt | Skills（Matt Pocock） |
|------|-------------------|----------------------|
| 格式 | Markdown | Shell Script |
| 复用性 | 复制粘贴 | Git Clone |
| 可调试性 | 难 | 容易（看源码） |

| 可测试性 | 无法测试 | Shell 测试 |
| 分享方式 | 截图/博客 | Git PR |

## 与上周博客的关联

Mario Zechner 上周发文《写了17年开源代码，我为什么认为 Coding Agents 堆功能是在瞎折腾？》，他的 `pi-mono` 项目就是这篇博客观点的**工程实现**。

**博客核心观点**：
- 不需要文件工具、子 agent、联网搜索
- 4 个工具（read/write/edit/bash）就够了
- 极短 system prompt
- 完全透明，无后台注入

**pi-mono 的实现**：

| 博客观点 | pi-mono 实现 |
|----------|--------------|
| 4 个工具 | TS 版 read/write/edit/bash |
| 极短 system prompt | 最短 prompt，模型自己知道 |
| Extension 机制 | 可扩展 skills、自定义工具 |
| 透明可观测 | 树形 session 结构 |

---

## 架构设计

```
pi-mono/
├── cli/                    # Agent CLI（TS）
│   ├── tools/             # 4 个核心工具
│   ├── session/           # 树形 session
│   └── llm/               # 统一 LLM 接口
├── python/                # 业务逻辑（Python）
│   ├── skills/            # Skills 实现
│   └── extensions/       # 扩展机制
├── ui/                   # TUI & Web UI
└── integrations/          # Slack Bot、vLLM 等
```

---

## 关键创新

1. **TypeScript + Python 混合** — Agent 核心用 TS（类型安全），业务逻辑用 Python（生态丰富）
2. **统一 LLM API** — 一个接口切换多个模型，不被供应商绑定
3. **树形 Session** — 支持分支和回溯，不是线性聊天
4. **Extension 机制** — Skills 作为扩展，不是内置功能

# 详细分析 #3：`mksglu/context-mode` — 上下文窗口的工程优化

## 基本信息

| 指标 | 数值 |
|------|------|
| **总 Stars** | 10,854 |
| **本周 Stars** | 2,346 |
| **主要语言** | TypeScript |

---

## 解决的问题

AI Coding Agent 的上下文窗口是稀缺资源。常见的浪费：
- 工具输出全量进入上下文
- 大量重复的系统 prompt
- 历史消息无差别堆放

## 解决方案

`context-mode` 的核心思路：**沙盒化工具输出**。

不是把工具输出全部扔进上下文，而是：
1. 工具在沙盒中执行
2. 只把"关键结果"传回上下文
3. 完整输出存在外部，按需检索

**效果**：减少 98% 的上下文占用。

---

## 支持平台

- Claude Code
- Cursor
- Copilot
- ...（共 14 个）

---

## 与 Garry Tan Resolver 的对比

| 维度 | context-mode | Garry Tan Resolver |
|------|-------------|-------------------|
| 目标 | 减少上下文占用 | 加载正确上下文 |
| 方法 | 沙盒化 + 选择性传回 | 路由表 |
| 层次 | 执行层优化 | 架构层设计 |
| 透明度 | 高（沙盒隔离） | 高（显式路由） |

两者是正交的——可以同时使用。
